    Generalidades BD

    Reglas de estandarización para el diseño y formato de la Base de Datos en Mermaid:

    Convenciones Generales

    1. Nomenclatura de Tablas:
    2. Utilizar notación PascalCase (también conocida como UpperCamelCase) y nombres descriptivos para las tablas.

    3. El prefijo "B_" puede ser usado para tablas base y "R_" para tablas de relación, asegurando que este prefijo se mantenga consistente en toda la base de datos.

    4. Nomenclatura de Columnas:

    5. Usar snake_case para los nombres de las columnas.

    6. Los nombres deben ser descriptivos y reflejar claramente el contenido de la columna.

    7. Tipos de Datos:

    8. Definir y estandarizar el uso de tipos de datos para ciertos tipos de información, como DATETIME para fechas y INT para identificadores.

    9. Utilizar ENUM para campos con un conjunto limitado de valores posibles, definiendo claramente los valores permitidos.

    10. Claves Primarias (PK):

    11. Nombrar las claves primarias como nombreTabla_id en snake_case.

    12. Utilizar el tipo int para las claves primarias, y considerar si se requiere que sean autoincrementales.

    13. Claves Foráneas (FK):

    14. Nombrar las claves foráneas con el formato nombreReferencia_id en snake_case.
    15. Incluir la referencia exacta en la declaración de la clave foránea, utilizando la sintaxis [ref: > NombreTablaReferenciada.nombreColumnaReferenciada].

    Reglas Específicas

    1. ENUMs:
    2. Definir claramente los valores ENUM al inicio del diseño de la base de datos y mantener la consistencia en las tablas.

    3. Considerar el uso de tablas de referencia para estados en lugar de ENUMs si se espera que la lista de estados crezca o cambie con el tiempo.

    4. Campos Booleanos:

    5. Utilizar nombres que indiquen claramente que la columna es de tipo booleano, como is_service, is_price_change_allowed, etc.

    6. Asegurarse de que los campos booleanos no contengan otros valores aparte de TRUE/FALSE o equivalentes numéricos.

    7. Campos de Estado:

    8. Usar campos ENUM para los estados con un conjunto limitado y conocido de valores, por ejemplo, status ENUM('1-active', '2-inactive', '3-deleted').

    9. Campos de Fecha:

    10. Utilizar DATETIME para todas las columnas que almacenen fechas y horas.

    11. Nombrar las columnas relacionadas con fechas de manera consistente, por ejemplo, date_created y date_updated.

    12. Consistencia en Campos Relacionados:

      • Asegurarse de que los campos relacionados a través de claves foráneas tengan el mismo tipo de dato y restricciones en ambas tablas.

    Convenciones de Documentación

    1. Comentarios y Documentación:
      • Documentar cada tabla y columna con comentarios que expliquen su propósito y cualquier regla de negocio asociada.
      • Incluir restricciones y reglas de negocio como comentarios en las definiciones de tabla cuando sea apropiado.

    Al aplicar estas reglas, se garantiza una mayor coherencia, legibilidad y mantenibilidad en el diseño de la base de datos. Además, se facilita el entendimiento y colaboración entre diferentes desarrolladores y analistas de datos que trabajen con la base de datos.
